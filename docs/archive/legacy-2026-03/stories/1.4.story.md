```typescript
/**
 * Parse command-line arguments string into array.
 * Handles quotes, spaces, and escape characters.
 * 
 * @param argsString - Raw arguments string from configuration
 * @param logger - Optional VS Code logger for warnings (default: console)
 * @returns Array of individual arguments
 * 
 * @example
 * parseArgumentsString('--yolo') // ['--yolo']
 * parseArgumentsString('--flag "value with spaces"') // ['--flag', 'value with spaces']
 */
export function parseArgumentsString(
  argsString: string, 
  logger?: { warn: (message: string) => void }
): string[];
```

## Notes

### Refactoring Principles

1. **单一职责原则**: 参数解析是独立的职责，应该有独立的模块
2. **可测试性**: 独立模块更容易编写单元测试，不依赖 VS Code 环境
3. **可维护性**: 小模块更容易理解和修改
4. **向后兼容**: 保持所有现有功能行为不变

### Implementation Tips

1. **复制粘贴策略**: 先完全复制函数实现，确保行为一致
2. **逐步验证**: 每个步骤后运行编译和测试
3. **保留原函数**: 在删除前可以临时注释，确认新模块工作正常
4. **测试优先**: 先写单元测试，用测试验证迁移正确性

### Known Risks

- **导入路径**: 确保相对路径正确（`./argument-parser` 不是 `../argument-parser`）
- **类型推断**: 确保导出函数的类型被正确推断
- **日志行为**: 确认传递 `log` 对象后警告消息仍正常输出

### Benefits

1. **减少 extension.ts 复杂度**: 从 1,055 行减少到约 970 行
2. **提高测试覆盖**: 可以为参数解析编写独立单元测试
3. **代码复用**: 未来其他模块可能也需要参数解析功能
4. **维护性提升**: 修改参数解析逻辑不需要修改主扩展文件

### Future Opportunities

- 考虑将更多工具函数抽离（如终端状态管理、配置读取等）
- 建立 `src/utils/` 目录统一管理工具模块
- 添加更多单元测试覆盖其他复杂逻辑
